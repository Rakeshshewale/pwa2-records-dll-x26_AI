1. Executive Summary

This document covers the end-to-end design, development, and deployment of a Progressive Web App (PWA) for personal expense tracking. The app modernizes an existing 2-year-old expense tracker built with HTML form dropdowns and Google Sheets, replacing all inputs with a tap-first mobile UI while preserving complete backward compatibility with the existing Google Sheets backend and dashboard SUMIF formulas.

The project was built iteratively through human-AI collaboration over a single session, with real-time testing and debugging on a live mobile device.

Key achievement: The most common expense entry (UPI food purchase) requires only 2 taps + typing the amount + pressing Enter. The previous version required 5 dropdown selections.

2. Design Goals & Constraints

2.1 Primary Objectives

GoalApproach
Speed of entryButtons replace all dropdowns; smart defaults auto-select most common options
Zero disruptionRun side-by-side with existing app; same Google Sheet, same column structure
Mobile-firstOptimized for one-handed phone use with touch-friendly buttons
Offline capablePWA with service worker caching for low-connectivity areas
Visual insightsDaily view and monthly summary with charts built into the app

2.2 Hard Constraints

  • All category/subcategory strings must be character-perfect matches to existing Google Sheet SUMIF formulas
  • The Google Apps Script POST endpoint and authentication remain unchanged
  • The Google Sheet column structure (A–O) is frozen
  • No external databases — Google Sheets is the single source of truth

2.3 UX Principles

  • Tap, don't type — buttons replace all dropdowns
  • Smart defaults — most common selections auto-highlight on type selection
  • No interruptions — subtle "Saved ✅" below submit, auto-clears in 1.4s
  • Static layout — no swipeable rows, no collapsible sections
  • Glassmorphism UI — modern frosted-glass aesthetic with color-coded buttons

3.2 Data Flow

Write path (Entry → Sheet): User taps buttons → enters amount → submit fires POST with URLSearchParams → Apps Script validates auth → maps category to correct column (E–N) → appends row to sheet.

Read path (Sheet → Daily/Summary): Frontend sends GET with action + date/month + auth → Apps Script reads sheet, filters rows → reverse-maps columns E–N to category/categoryValue → returns JSON → frontend calculates balances and renders charts.

4.2 Category-per-Column Design

Key design decision: The sheet uses one column per category (E–N) rather than a single "category" column. Only one of these columns has a value per row. This supports the Dashboard sheet's SUMIF formulas referencing specific column headers.

doPost() writes:

const categoryIndex = categoryColumns.indexOf(e.parameter.category);
if (categoryIndex !== -1) {
  row[categoryIndex + 4] = e.parameter.categoryValue || '';
}

doGet() reads (reverse mapping):

for (var i = 0; i < categoryColumns.length; i++) {
  var val = row[i + 4];
  if (val && val.toString().trim() !== "") {
    category = categoryColumns[i];
    categoryValue = val.toString().trim();
    break;
  }
}

7. Balance Calculation Engine

7.1 Final Balance Formulas

HDFC = Opening + Salary − Salary to Kotak − Salary to ICICI − Cash withdrawal − CC bill payments
Kotak = Opening + Salary to Kotak + ICICI to Kotak + Other income − Kotak to ICICI − all UPI expenses
ICICI = Opening + Salary to ICICI + Kotak to ICICI − ICICI to Kotak − Investment
Wallet = Opening + E-wallet topup − all Wallet expenses
Cash = Opening + Cash withdrawal − all Cash expenses
Salary Available = Salary − Expenses − Investment − CC Bills Paid
Total = HDFC + Kotak + ICICI + Wallet + Cash

7.2 CC Next Bill (3rd–3rd Cycle)

Bill generation date is the 2nd, so expenses from the 3rd onwards count toward the next bill:

// Credit card expenses from 3rd onwards
if (mode === "Credit card" && dt.getDate() >= 3) ccFrom3rd += amt;

// E-wallet topup is also a CC charge
if (cat === "Money transfer" && det === "E-wallet topup" && dt.getDate() >= 3)
  ccFrom3rd += amt;

8.3 Monthly Trend Chart

Fetches 6 months of aggregated data via the trend backend action. Displays:

  • Red bars — Total expenses per month
  • Steel blue bars — CC bill payments per month
  • Dashed red line — Linear regression trend line showing expense direction

Y-axis shows values in ₹K/₹L format with subtle gridlines. No data labels on bars — exact values visible via tap/hover tooltip. The trend line is calculated client-side using linear regression:

function linReg(data) {
  var n = data.length, sx=0, sy=0, sxy=0, sx2=0;
  for (var i=0; i<n; i++) {
    sx+=i; sy+=data[i]; sxy+=i*data[i]; sx2+=i*i;
  }
  var slope = (n*sxy - sx*sy) / (n*sx2 - sx*sx);
  var intercept = (sy - slope*sx) / n;
  return data.map((_, i) => Math.round(slope*i + intercept));
}

8.4 Dynamic Bar Label Positioning (Category Chart)

Bars ≥60% of the maximum value show white text inside. Smaller bars show dark text outside. This prevents the largest bar's value from being pushed off-screen.

align: function(ctx) {
  var max = Math.max.apply(null, ctx.dataset.data);
  return ctx.dataset.data[ctx.dataIndex] >= max * 0.6 ? 'start' : 'right';
},
color: function(ctx) {
  var max = Math.max.apply(null, ctx.dataset.data);
  return ctx.dataset.data[ctx.dataIndex] >= max * 0.6 ? '#ffffff' : '#334155';
}

8.4 Indian Number Formatting

  return '₹' + Number(n).toLocaleString('en-IN');

function fmtK(n) {
  return n >= 100000 ? '₹' + (n/100000).toFixed(1) + 'L'

}

12. Google Apps Script Backend

12.1 doPost() — Write (Existing, Unchanged)

Receives 7 fields via URLSearchParams: authToken, transactionType, amount, expenseMode, category, categoryValue, additionalInfo. Maps category to column index, appends row.

12.2 doGet() — Read (New)

Two actions supported:

ActionParametersReturns
dailydate=YYYY-MM-DDAll rows matching that date
monthdatamonth=N&year=YYYYAll rows for that month
trendmonth=N&year=YYYY&count=6Monthly totals (expenses + CC bills) for last 6 months

12.3 Row Mapper (_mr)

Scans columns E–N (index 4–13), finds the first non-empty cell, uses its column header as category and cell value as categoryValue.

13. Debugging Log — 8 Bugs Found & Fixed

Bug 1: Opening Balance Overwrite
Multiple Cash entries (₹2K + ₹5K + ₹500) showed as ₹500. Used = amt instead of += amt.
Fix: Changed all opening balance assignments to accumulate: bal.Cash += amt
Bug 2: transactionType vs category Mismatch
Sheet stores "Money transfer" for both transfers AND opening balances. Code checked type === "Opening balance" which never matched.
Fix: Changed to cat === "Opening balance" — uses the category field from doGet() reverse mapping.
Bug 3: UPI Mode Persisting for Transfers
Opening balance entries showed "UPI" as payment mode because auto-select remained in state.
Fix: Clear S.mode and remove button highlight when switching away from Expense type.
Bug 4: Frozen UI After Update
No buttons clickable. Second rows.forEach in calcSummary referenced cat variable not defined in that scope.
Fix: Added const cat = r.category || '' to the second loop.
Bug 5: Bar Chart Value Hidden
Largest bar pushed its value label off-screen.
Fix: Dynamic positioning — bars ≥60% of max show white text inside, others show dark text outside.
Bug 6: "Electronics" Text Wrapping
5 buttons per row caused long words to wrap onto 2 lines.
Fix: Changed all detail rows to 3 per row for consistency.
Bug 7: Salary to Kotak/ICICI Not Deducting HDFC
HDFC balance too high — transfers out weren't deducting from source.
Fix: Added bal.HDFC -= amt for both Salary to Kotak and Salary to ICICI.
Bug 8: Smart Quotes in Apps Script
Copying code from AI artifact converted straight quotes to curly quotes, causing syntax errors.
Fix: Paste into Notepad first, or manually retype quotes in the script editor.

16. Inputs for Future AI Models

Critical context for any AI continuing work on this project:
  1. String exactness matters. "Kotak Balance" has capital B. "Other income" has lowercase i. Every value must match Google Sheet SUMIF formulas.
  2. transactionType ≠ category. Sheet column C stores "Money transfer" for both transfers and opening balances. Always check the category field to distinguish.
  3. Opening balances accumulate. Multiple entries per account must ADD (+=), not overwrite (=).
  4. Payment mode scope. expenseMode only applies to Expense type. Must be empty for Transfer and CC Bill.
  5. Column structure is frozen. Columns A–O cannot change. Dashboard has SUMIF formulas on column headers.
  6. Google Apps Script quirks. Smart quotes from web break syntax. Use var for compatibility. New deployment version required for each change.
  7. CC bill cycle is 3rd to 3rd. Includes both Credit card mode expenses AND E-wallet topup transfers.
  8. HDFC is the hub. Salary lands in HDFC, then distributes. All CC bills, cash withdrawals, and bank transfers deduct from HDFC.
  9. Salary breakup includes CC bills. The pie chart has 4 slices: Investment, Expenses, CC Bill Paid, Available. Available = Salary − Expenses − Investment − CC Bills. CC bills are paid from HDFC (salary account) so they are part of the salary utilization.
  10. Local demo store removed. Daily and Summary tabs are now pure API-only. They require the doGet() backend to function.
  11. Chart.js datalabels. Must register globally: Chart.register(ChartDataLabels). Dynamic positioning via callback functions.

Testing Checklist

  • Submit an Expense with UPI — verify correct column gets subcategory value
  • Submit a Transfer — verify expenseMode is empty in sheet
  • Submit a CC Bill — verify additionalInfo has the card name
  • Check Daily tab — entries appear with correct colors
  • Check Summary tab — balances match manual calculation
  • Switch between Expense/Transfer/CC Bill — no stale state carried over
  • Clear button resets everything including mode highlights
  • Enter key on amount field triggers submit
  • Opening balance with multiple entries — verify they add up

Known Limitations

  1. No edit or delete for submitted entries
  2. No multi-month trend analysis
  3. No budget setting or alerts
  4. Auth token is plaintext in frontend
  5. Full sheet scan on every doGet() — no indexing
  6. Service worker may cache stale JS — users need hard refresh
  7. No data export from PWA
